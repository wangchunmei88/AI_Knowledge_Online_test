Qwen 系列（通义千问 - 阿里）

Alibaba 2023 - 至今 Decoder-only

📌 一句话定位

Qwen = 国产大模型的「全能选手」
从语言到视觉到代码，从 0.5B 到 72B+，覆盖全尺寸全模态，开源生态最完善的国产大模型之一。

🏗️ 核心架构

# Qwen 架构核心（与 LLaMA 高度相似） class QwenBlock: def forward(x): # 1. Grouped Query Attention + RoPE x = x + GQA(RoPE(LN(x))) # 2. SwiGLU FFN x = x + SwiGLU(LN(x)) return x

💡 Qwen 的三大亮点

  1. 超大词表（~150K）：远大于 LLaMA 的 32K/128K，对中文和多语言支持更好，中文 token 压缩率更高
  2. Tie Embedding（词嵌入绑定）：Qwen-2 起输入嵌入和输出 lm_head 共享权重，减少参数量
  3. Dual Chunk Attention（Qwen-2 128K）：将长序列分成 chunk，分块处理注意力，支持 128K 超长上下文

🔷 Qwen-VL（视觉语言模型）

Qwen 的多模态版本，支持图像输入：

  • Vision Encoder：ViT-bigG（约2B参数），将图像编码为 token 序列
  • 适配层：将视觉 token 压缩后送入 LLM
  • 多图理解：支持多张图片输入 + OCR

🎯 面试常考点

  • Qwen 和 LLaMA 架构上的主要区别？
    → 架构高度相似（RoPE + RMSNorm + SwiGLU + GQA），区别在于词表大小（Qwen 150K vs LLaMA 32K/128K）、Tie Embedding、YaRN 长度外推
  • 为什么 Qwen 的词表这么大？
    → 中文 token 种类多，大词表让中文编码更高效，单 token 包含更多信息，推理更快

GLM 系列（智谱AI）

Zhipu AI 2022 - 至今 Prefix-Decoder

📌 一句话定位

GLM = 唯一使用 Prefix-Decoder 架构的主流大模型
自创的 GLM 预训练范式，将理解（编码器）和生成（解码器）统一在同一个模型中，是国产大模型中架构最独特的。

🏗️ 核心创新：Prefix-Decoder

GLM 最大的特色是其独特的注意力模式：

# 三种注意力模式对比 # 1. Encoder（BERT）: 双向注意力 # 每个 token 可以看到所有其他 token [A→B, A→C, A→D] # A 可以看 B, C, D # 2. Decoder（GPT）: 单向因果注意力 # 每个 token 只能看到左边的 token [A→A, B→A, C→AB, D→ABC] # D 只能看 A, B, C # 3. Prefix-Decoder（GLM）: 混合注意力 # Prefix 部分（如 Prompt）：双向注意力 # Generation 部分：因果注意力 + 能看到 Prefix # Part 1 (Prefix): 可以互相看 → 双向 # Part 2 (Generation): 只能看左边 + 全部 Prefix → 单向+双向混合
通俗理解：
· Prompt 部分 = "大家互相认识"（双向注意力，充分理解上下文）
· 生成部分 = "只能看到之前说的 + 听到 Prompt"（因果注意力，自回归生成）

这样 GLM 兼顾了理解能力（类似 BERT）和生成能力（类似 GPT）。

💡 GLM 的独特之处

  1. 唯一主流 Prefix-Decoder：其他模型要么纯 Encoder（BERT），要么纯 Decoder（GPT/LLaMA），只有 GLM 走了混合路线
  2. 自创预训练范式：GLM 论文提出"自回归空白填充"（Autoregressive Blank Infilling），将理解和生成统一
  3. 国产最早开源之一：ChatGLM-6B 是国内最早的一批开源对话模型

📝 自回归空白填充

# GLM 预训练任务 原文: "今天天气很好，[M]，适合出门散步" # Step 1: 随机遮盖一段文本 → [M] # Step 2: 模型自回归地生成被遮盖的内容 预测: "阳光明媚" # 自回归生成 # 对比 BERT: 用 [MASK] 替换，但不是自回归的 # 对比 GPT: 从左到右预测下一个词，没有遮盖 # GLM: 结合两者，遮盖 + 自回归生成

🎯 面试常考点

  • GLM 和 GPT 的架构区别？
    → GLM 用 Prefix-Decoder：Prompt 部分双向注意力，生成部分因果注意力；GPT 全部因果注意力
  • GLM 的预训练任务是什么？
    → 自回归空白填充（Autoregressive Blank Infilling），遮盖一段文本后自回归生成
  • Prefix-Decoder 有什么优势？
    → 兼顾理解（双向看 Prompt）和生成（因果生成），一个模型同时擅长 NLU 和 NLG

Claude 系列（Anthropic）

Anthropic 2023 - 至今 Decoder-only

📌 一句话定位

Claude = 以「安全对齐」和「超长上下文」著称的闭源大模型
由 OpenAI 前员工创立的 Anthropic 开发，核心特色是 Constitutional AI（宪法AI）对齐方法和 200K+ 超长上下文。

🔑 核心创新：Constitutional AI（宪法AI）

核心思想：用一组「宪法原则」指导 AI 自我改进，减少对人类标注的依赖，同时提升安全性。

训练流程：

  1. 监督学习阶段：模型生成回答 → 用宪法原则自我评估 → 生成修正后的回答 → 用修正数据微调
  2. 强化学习阶段：用微调后的模型作为 RM（奖励模型），根据宪法原则打分 → 用 RL 进一步优化
# Constitutional AI 核心流程 # 宪法原则示例： 原则1: "选择最无害且最有帮助的回答" 原则2: "不要提供危险或非法的指导" 原则3: "承认不确定性，不要编造事实" # 自我评估流程： 1. 模型生成两个回答 A 和 B 2. 模型根据宪法原则评估哪个更好 3. 用评估结果生成训练数据 4. 微调模型使其更符合宪法原则

💡 Claude 的三大特色

  1. 超长上下文（1M token）：Claude 4.1 Opus 支持 100 万 token 上下文，业界最长，适合超长文档/代码库分析
  2. 安全对齐：Constitutional AI 让 Claude 在安全性上表现出色，减少有害输出
  3. 动态推理（Extended Thinking）：Claude 4 起支持"扩展思考"模式，模型可以在回答前进行内部推理链，类似 o1/R1

📝 Claude 架构关键细节

与 LLaMA/Qwen 的架构差异：
· 注意力：Claude 用 MHA（多头注意力），而非 GQA（分组查询注意力）
· 位置编码：Claude 用绝对位置编码（可学习），而非 RoPE（旋转位置编码）
· 动态推理：Claude 4+ 支持 Extended Thinking，在回答前可以"内部思考"多步
· 1M 上下文：通过改进的位置编码外推和高效的注意力实现支持超长序列

🎯 面试常考点

  • Constitutional AI 和 RLHF 的区别？
    → RLHF 需要人类标注偏好数据；CAI 用宪法原则让模型自我评估，减少人工标注，可扩展性更强
  • Claude 为什么上下文能做那么长？
    → 可能的优化：改进的位置编码外推、高效的注意力实现、训练时使用长文本数据

Gemini 系列（Google）

Google DeepMind 2023 - 至今 多模态原生

📌 一句话定位

Gemini = Google 的「原生多模态」大模型
从一开始就设计为多模态模型，而非后接视觉编码器。文本、图像、音频、视频使用同一套 Transformer 架构处理。

🏗️ 核心架构

# Gemini 架构核心 # 原生多模态：不同模态映射到同一 token 空间 文本 token → Token Embedding → ─┐ 图像 patch → ViT Encoder → ─┤ 音频 frame → Audio Encoder → ─┼→ 统一 Transformer → 输出 视频帧 → 时空编码器 → ─┤ # 关键：所有模态在同一个 Transformer 中处理 # 而不是 "LLM + 外接视觉编码器" 的拼接方式

💡 Gemini 的三大亮点

  2. 200万 token 上下文：Gemini-1.5 Pro 起支持 2M 上下文，可以一次读完一整本书+一部电影
  3. Gemma 开源系列：Google 基于 Gemini 技术推出的开源模型，最新 Gemma 4（2026.4） 已发布，Apache 2.0 协议

🎯 高频面试 Q&A

Prefix-Decoder 和纯 Decoder-only 有什么区别？GLM 为什么要用 Prefix-Decoder？
区别：
· 纯 Decoder：所有 token 都是因果注意力（只能看左边）
· Prefix-Decoder：Prompt 部分双向注意力，生成部分因果注意力

GLM 选择的原因：
· Prompt 部分用双向注意力 → 对用户输入理解更充分（类似 BERT）
· 生成部分用因果注意力 → 自回归生成（类似 GPT）
· 一个模型同时擅长理解和生成，不需要像 BERT+GPT 那样用两个模型
Qwen 为什么用 150K 的大词表？词表大小对模型有什么影响？
大词表的优势：
· 中文 token 种类多，大词表让更多中文词直接映射为单个 token
· 单 token 信息密度更高 → 同样内容用更少 token → 推理更快
· 多语言支持更好，中英日韩等语言编码效率都更高

大词表的代价：
· 词表越大，embedding 层参数越多（150K × 4096 ≈ 600M 参数仅用于 embedding）
· Softmax 计算量增大（在 150K 个词上计算概率）

结论：对中文为主的场景，大词表的收益大于代价。
Constitutional AI 和 RLHF 有什么区别？各有什么优缺点？
RLHF：
· 流程：模型生成多个回答 → 人类标注偏好 → 训练奖励模型 → PPO 强化学习
· 优点：直接对齐人类偏好
· 缺点：需要大量人工标注，成本高，难以扩展

Constitutional AI：
· 流程：模型生成回答 → 用「宪法原则」让模型自我评估 → 生成修正数据 → 微调
· 优点：减少人工标注，可扩展性强，安全性更可控
· 缺点：宪法原则的设计需要专业知识，自我评估可能存在偏差

关键区别：CAI 用 AI 评估替代人类标注，RLHF 用人类标注。
Claude 用的是 MHA + 绝对位置编码，而 LLaMA 用 GQA + RoPE，这两种方案各有什么优劣？
MHA vs GQA：
· MHA：每个头有独立的 K、V，效果最好但 KV Cache 占用大
· GQA：多组 Q 共享 K、V，KV Cache 更小，推理更快
· Claude 选 MHA → 追求效果，通过其他方式（如更好的基础设施）解决显存问题

绝对位置编码 vs RoPE：
· 绝对位置编码：为每个位置分配唯一编码，训练简单但外推性有限
· RoPE：编码相对位置，外推性好，主流开源模型标配
· Claude 选绝对位置编码 → Anthropic 可能有改进的外推技术（如位置插值），使绝对编码也能支持 1M 上下文

总结：Claude 的选择更"保守"（用标准 MHA + 绝对位置编码），但通过工程优化实现了 1M 上下文和动态推理等高级功能。
