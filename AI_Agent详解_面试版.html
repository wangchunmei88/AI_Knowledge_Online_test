❓ 什么是 AI Agent

一句话定义

AI Agent = 大模型 + 记忆 + 工具 + 规划
能够自主感知环境、制定计划、调用工具、执行任务，并根据反馈调整行为的智能体。

Agent 与普通 LLM 的本质区别

维度 普通 LLM AI Agent
交互方式 一问一答，被动响应 多轮自主决策，主动执行
工具使用 ❌ 无法调用外部工具 ✅ 可调用搜索、代码、API等
记忆能力 仅限上下文窗口 短期+长期记忆，跨会话持久化
规划能力 无，直接生成回答 拆解任务、制定计划、逐步执行
错误处理 无法自我纠正 观察结果→反思→调整策略

形象比喻

LLM = 一个博学但只会说话的人，你问什么他答什么
Agent = 一个有手有脚的助手，不仅能回答问题，还能帮你查资料、写代码、发邮件、操作软件

🧠 Agent 核心架构

四大核心组件

👤 用户输入 → 🧠 LLM（大脑） ↔ 📋 规划（Planning）

🧠 LLM（大脑） ↔ 💾 记忆（Memory） ↔ 🔧 工具（Tools）

🧠 LLM（大脑） → 📤 执行动作 / 返回结果

1️⃣ LLM（大脑）

Agent 的核心推理引擎，负责理解意图、决策、生成回复。

  • 作用：理解用户需求 → 选择工具 → 解析结果 → 决定下一步
  • 关键能力：指令遵循、推理、函数调用（Function Calling）
  • 常见选择：GPT-4、Claude、Qwen、DeepSeek 等

2️⃣ 规划（Planning）

将复杂任务拆解为可执行的子步骤：

# 用户：帮我分析竞品并写一份报告 # Agent 的规划过程： Step 1: 搜索竞品A的最新信息 → 调用搜索工具 Step 2: 搜索竞品B的最新信息 → 调用搜索工具 Step 3: 整理对比数据 → LLM 推理 Step 4: 生成分析报告 → LLM 生成 Step 5: 保存到文档 → 调用文档工具

3️⃣ 记忆（Memory）

📝 短期记忆

当前对话的上下文历史，存在 LLM 的上下文窗口中

例如：之前3轮对话的内容

💾 长期记忆

持久化存储的用户偏好、历史交互、知识库

例如：用户喜欢中文回复、上次查询的结果

4️⃣ 工具（Tools）

Agent 与外部世界交互的接口：

工具类型 示例
🔍 搜索 Google Search、Bing Search、Tavily
💻 代码执行 Python REPL、沙箱环境
📁 文件操作 读写文件、操作数据库
🌐 API 调用 天气API、股票API、邮件API
🤖 其他Agent 调用子Agent处理特定子任务

🎯 面试常考点

  • Agent 和 LLM 的区别？
    → Agent = LLM + 记忆 + 工具 + 规划，能自主决策和执行，不只是被动回答
  • Agent 的四大核心组件？
    → LLM（大脑）、Planning（规划）、Memory（记忆）、Tools（工具）

🔄 Agent 设计模式

1️⃣ ReAct 模式（最经典）

ReAct = Reasoning + Acting
交替进行「思考」和「行动」，每一步都先推理再执行，再根据结果继续推理。
💭 Thought（思考） → ⚡ Action（行动） → 👁️ Observation（观察） → 💭 Thought → ...循环
# ReAct 示例：查询北京天气并建议穿搭 Thought 1: 我需要先查北京今天的天气 Action 1: call weather_api(city="北京") Observation 1: 北京今天 8°C，多云，北风3级 Thought 2: 8°C比较冷，需要保暖穿搭建议 Action 2: 无需额外工具，直接根据天气给出建议 Answer: 北京今天8°C多云偏冷，建议穿厚外套+毛衣+长裤，记得带围巾防风。

2️⃣ Reflexion 模式（自我反思）

在 ReAct 基础上增加自我评估和反思能力：

⚡ 执行任务 → 🔍 自我评估 → 💭 反思改进 → ⚡ 重新执行

核心：Agent 完成任务后，自我评估结果质量，如果不好就反思原因，重新尝试。

3️⃣ Plan-and-Execute 模式

先完整规划，再逐步执行：

# Plan 阶段：一次性生成完整计划 Plan: 1. 搜索竞品A信息 2. 搜索竞品B信息 3. 对比分析 4. 生成报告 # Execute 阶段：按计划逐步执行 Execute Step 1 → 结果1 Execute Step 2 → 结果2 Execute Step 3 → 结果3 # 可能根据结果调整计划 Execute Step 4 → 最终报告

对比 ReAct：ReAct 每步都重新思考；Plan-and-Execute 先做全局规划，适合复杂多步任务。

4️⃣ Multi-Agent 模式（多智能体协作）

🔄 串行协作

Agent A 完成 → 传给 Agent B → 传给 Agent C

如：研究员→写手→审核员

🤝 并行协作

多个 Agent 同时处理不同子任务

如：3个研究员分别搜索3个方向

👑 主从协作

一个主 Agent 分配任务给多个子 Agent

如：经理分配给设计师、开发、测试

🗣️ 辩论协作

多个 Agent 各自独立推理，投票/协商决策

如：3个Agent各自分析后投票

🎯 面试常考点

  • ReAct 的核心流程？
    → Thought → Action → Observation 循环，交替推理和行动
  • ReAct 和 Plan-and-Execute 的区别？
    → ReAct 每步重新思考；Plan-and-Execute 先全局规划再执行，适合复杂任务
  • Multi-Agent 有哪几种协作模式？
    → 串行、并行、主从、辩论

🔧 工具调用（Function Calling）

核心原理

Function Calling = 让大模型输出结构化的「函数调用指令」，而不是纯文本回复。
模型本身不执行函数，而是告诉你「该调用什么函数、传什么参数」，由你的代码去执行。

工作流程

👤 用户提问 → 🧠 LLM 判断：需要调用工具？

✅ 是：生成函数调用JSON → ⚙️ 你的代码执行函数 → 📤 结果返回LLM → 💬 最终回答

❌ 否：直接生成文本回答

Function Calling 示例

// 1. 定义工具（告诉模型有哪些工具可用） tools = [{ "name": "get_weather", "description": "查询指定城市的天气", "parameters": { "city": { "type": "string", "description": "城市名" } } }] // 2. 模型返回：不是文本，而是函数调用指令 { "tool_calls": [{ "function": { "name": "get_weather", "arguments": { "city": "北京" } } }] } // 3. 你的代码执行 get_weather("北京") → 返回结果给模型 // 4. 模型根据结果生成最终回答

关键细节

  • 模型不执行函数：它只输出「调用什么、传什么参」，执行由你的代码完成
  • 如何判断需要工具：模型根据工具的 description 自主判断
  • 多工具选择：可以一次定义多个工具，模型自主选择合适的
  • 并行调用：模型可以一次返回多个函数调用（如同时查天气和股票）

🎯 面试常考点

  • Function Calling 的原理？
    → 模型输出结构化 JSON 指令，由外部代码执行，结果返回模型继续推理
  • 模型本身会执行函数吗？
    → 不会！模型只决定「调用什么」，执行由开发者代码完成
  • 如何让模型知道有哪些工具可用？
    → 通过 tools 参数传入工具的名称、描述、参数定义

💾 Agent 记忆系统

记忆层次

类型 存储位置 持久性 示例
工作记忆 LLM 上下文窗口 当前对话 最近几轮对话历史
短期记忆 会话缓存 一次会话 当前任务中间结果
长期记忆 向量数据库 / KV存储 跨会话持久 用户偏好、历史摘要

长期记忆的实现方案

📋 摘要记忆

定期对对话做摘要，压缩后存储。节省空间但可能丢失细节。

🔍 向量检索记忆

将历史交互向量化存入向量库，需要时检索相关片段。灵活但需要额外基础设施。

🗂️ 知识图谱记忆

抽取实体和关系构建图谱，支持复杂推理。结构化但构建成本高。

🎯 面试常考点

  • 为什么 Agent 需要长期记忆？
    → LLM 上下文窗口有限（如128K），无法存储所有历史；跨会话需要持久化
  • 如何实现长期记忆？
    → 向量数据库 + 摘要 + 知识图谱，根据场景选择

LangChain 核心概念

# LangChain 构建 Agent 的核心组件 from langchain.agents import AgentExecutor, create_react_agent from langchain.tools import Tool from langchain.memory import ConversationBufferMemory # 1. 定义工具 tools = [ Tool(name="search", func=search_func, description="搜索信息"), Tool(name="calculator", func=calc_func, description="数学计算"), ] # 2. 配置记忆 memory = ConversationBufferMemory() # 3. 创建 Agent（使用 ReAct 模式） agent = create_react_agent(llm, tools, prompt_template) # 4. 执行器（管理 Agent 运行循环） agent_executor = AgentExecutor(agent=agent, tools=tools, memory=memory)

🎯 高频面试 Q&A

什么是 AI Agent？它和 LLM 的区别是什么？
定义：AI Agent 是能够自主感知环境、规划任务、调用工具、执行动作并根据反馈调整行为的智能体。

与 LLM 的区别：
· LLM 只能被动回答问题，Agent 能主动执行任务
· LLM 无法调用外部工具，Agent 可以使用搜索、代码、API等工具
· LLM 没有持久记忆，Agent 有短期+长期记忆系统
· LLM 无法自主规划，Agent 能拆解复杂任务并逐步执行

公式化：Agent = LLM + Memory + Tools + Planning
ReAct 模式的核心思想是什么？有什么优缺点？
核心思想：Reasoning + Acting 交替进行。每一步先思考（Thought），再执行动作（Action），观察结果（Observation），然后继续思考。

优点：
· 每步都有推理过程，决策可解释
· 可以根据中间结果灵活调整策略
· 减少幻觉（基于实际观察而非猜测）

缺点：
· 每步都调用 LLM，Token 消耗大
· 可能陷入循环（反复尝试同一错误策略）
· 对简单任务过于复杂，效率低
Function Calling 是怎么实现的？模型本身会执行函数吗？
实现原理：
1. 开发者定义工具的名称、描述、参数格式，传给模型
2. 用户提问后，模型判断是否需要调用工具
3. 如果需要，模型输出结构化 JSON（函数名+参数）
4. 开发者代码解析 JSON，执行对应函数
5. 将执行结果返回模型，模型生成最终回答

关键：模型本身不执行函数！它只是「决定调用什么」，执行由外部代码完成。这是很多面试官会追问的点。
Agent 的记忆系统如何设计？如何解决上下文窗口有限的问题？
三层记忆架构：
1. 工作记忆：当前对话历史，直接放在 LLM 的 Prompt 中
2. 短期记忆：当前任务的中间结果，缓存到内存中
3. 长期记忆：跨会话的持久化存储，用向量数据库检索

解决上下文有限的方法：
· 摘要压缩：对长对话做摘要，用摘要替代原始内容
· 滑动窗口：只保留最近 K 轮对话
· RAG 检索：将历史存入向量库，需要时检索相关片段
· 重要度筛选：只保留重要信息，丢弃冗余内容
Multi-Agent 系统有哪些协作模式？各适合什么场景？
四种模式：

1. 串行（Pipeline）：A→B→C 依次处理
→ 适合有明确流程的任务（研究→写作→审核）

2. 并行（Parallel）：多个 Agent 同时处理不同子任务
→ 适合可分解的独立子任务（同时搜索多个方向）

3. 主从（Hierarchical）：一个主 Agent 分配任务给子 Agent
→ 适合复杂项目（项目经理分配给开发、设计、测试）

4. 辩论（Debate）：多个 Agent 各自推理，投票决策
→ 适合需要多角度验证的任务（代码审查、方案评审）
Agent 在实际落地中面临哪些挑战？
五大核心挑战：

1. 可靠性：LLM 可能选错工具、生成错误参数，导致执行失败
→ 解法：增加校验层、人工确认关键操作

2. 成本控制：每步都调用 LLM，Token 消耗大
→ 解法：缓存常见模式、小模型做简单决策

3. 安全性：Agent 有执行权限，可能误操作（删文件、发邮件）
→ 解法：权限控制、沙箱环境、人工审批

4. 调试困难：多步执行+工具调用，出错难定位
→ 解法：详细日志、链路追踪

5. 评估困难：Agent 行为非确定性，难以量化评估
→ 解法：任务完成率、步骤准确性、人工评测
