为什么用 BF16 而不是 FP16？BF16 牺牲了尾数精度（7位 vs 10位），但保持了与 FP32 相同的指数范围（8位），因此不会溢出也不需要 Loss Scaling，训练更稳定。A100/H100 原生支持 BF16。

⚙️ SFT 训练细节

Loss 只计算回答部分

# SFT 的关键：loss 只在 response 部分计算！ # # 输入: [instruction_tokens] [response_tokens] # mask: [ 0 ] [ 1 ] # loss: 只计算 mask=1 的位置的预测损失 # 为什么？因为我们要模型学习"如何回答" # 而不是学习"指令长什么样" def sft_loss(logits, labels, mask): # labels: 真实token id # mask: 0=instruction, 1=response loss_per_token = cross_entropy(logits, labels) return (loss_per_token * mask).sum() / mask.sum()

03高效微调（PEFT）

🎯 为什么需要 PEFT？

全参数微调（Full Fine-Tuning）虽然效果最好，但在实际工程中面临三大核心痛点：

💾 显存开销巨大

微调一个 7B 模型，仅模型权重就需要 ~14GB（FP16），加上梯度、优化器状态，总计需要 40-60GB+ 显存。70B 模型更是需要多卡并行。

显存瓶颈

🧠 灾难性遗忘

全参微调会大幅改变预训练学到的通用知识，模型在下游任务上变强，但在其他任务上性能严重退化。学新忘旧，难以多任务兼顾。

知识丢失

📦 存储成本高

每个下游任务都需要存储一份完整的模型副本。100 个任务 = 100 份完整权重，存储开销线性增长。

存储爆炸
PEFT 核心思路：冻结预训练模型的大部分参数，只训练少量额外参数（通常 <1%），即可达到接近甚至等于全参微调的效果。训练快、显存省、每个任务只存小量适配器。

🔧 LoRA 详解

核心方法 面试高频

📌 核心思想：低秩分解

LoRA（Low-Rank Adaptation）的核心观察：模型微调时的权重变化矩阵 ΔW 具有低秩特性，可以用两个小矩阵的乘积来近似。

# 原始权重更新：W = W₀ + ΔW # LoRA 的低秩分解：ΔW = B · A # 其中 W₀ ∈ R^(d×k), B ∈ R^(d×r), A ∈ R^(r×k), r << min(d,k) def lora_forward(x, W0, A, B, alpha, r): # 原始预训练路径（冻结） h_pretrained = x @ W0 # LoRA 适配路径（可训练） h_lora = (x @ A.T) @ B.T # 缩放因子：alpha / r scaling = alpha / r return h_pretrained + scaling * h_lora # 示例：d=4096, k=4096, r=8 # 原始参数量：4096 × 4096 = 16,777,216 # LoRA 参数量：(4096 × 8) + (8 × 4096) = 65,536（仅 0.39%）
维度拆解：假设原始权重 W₀ ∈ ℝ4096×4096，取秩 r=8：
  • A ∈ ℝ8×4096（初始化为高斯分布）
  • B ∈ ℝ4096×8（初始化为零矩阵 → 训练开始时 ΔW=0）
  • 可训练参数：65,536，仅占原始参数的 0.39%

📐 秩 r 的选择

秩 r 是 LoRA 最关键的超参数，控制了适配器的表达能力和参数量：

秩 r参数量占比适用场景说明
4~0.2%简单任务、数据少表达力有限，但更不易过拟合
8~0.4%通用推荐起步值平衡效果与效率的甜点
16~0.8%中等复杂任务多数场景够用
32-64~1.5-3%复杂任务、数据充足接近全参微调效果
128+>5%极复杂任务收益递减，谨慎使用
⚠️ r 不是越大越好！r 过大可能导致过拟合（尤其数据量小时），且训练/推理开销增加。经验法则：先从 r=8 开始，根据验证集表现逐步调整。

🎛️ Alpha 参数（缩放系数）

LoRA 的输出会乘以缩放因子 alpha / r：

h = W₀ · x + (alpha / r) · B · A · x
  • alpha 的作用：控制 LoRA 适配的"强度"。alpha 越大，适配器的影响力越大。
  • 常见设置：alpha = 2×r（如 r=8 时 alpha=16），这样 scaling = 2.0
  • 调试技巧：调整 alpha 比调整 r 更高效——改 alpha 不影响参数量，只影响更新幅度

🧬 为什么 LoRA 有效？内在维度假说

LoRA 有效的理论支撑来自 Aghajanyan et al. (2020) 的研究：

内在维度假说（Intrinsic Dimensionality Hypothesis）：预训练模型已经学到了高质量的通用表示，微调时权重的有效变化只发生在一个低维子空间中。也就是说，ΔW 的"有效秩"远小于其数学维度。
# 直觉理解： # 预训练模型 W₀ 已经在参数空间中找到了一个好的区域 # 微调只需要在这个区域内做小幅调整 # 而这个调整可以用低秩矩阵充分表达 # 实验证据： # RoBERTa-Large 微调时，ΔW 的有效秩仅约 1-10 # GPT-3 175B 微调时，r=4 或 r=8 即可达到全参微调 99%+ 的效果

🎯 LoRA 应用在哪些层？

不同策略对效果和参数量有显著影响：

应用策略目标层参数量效果说明
仅 Q/VQuery & Value 投影最少良好LoRA 原始论文默认方案
Q/K/V/O全部注意力投影中等较好常见工程实践
全层 LoRA注意力 + FFN较多最好QLoRA 论文推荐方案
实践经验：QLoRA 论文（Dettmers et al. 2023）发现，在所有线性层上应用 LoRA（r=64）的效果优于仅应用于注意力层。原因是 FFN 层也包含大量可适配的知识，跳过它们会限制适配能力。

⚡ LoRA 的合并与推理

LoRA 最优雅的特性之一：训练完成后，可以将 LoRA 权重合并回基础模型，推理零开销。

# ========== 训练阶段 ========== # W₀ 保持冻结，只训练 A 和 B h = W₀ · x + (alpha/r) · B · A · x # ========== 合并阶段 ========== # 将 LoRA 权重合并到基础权重 W_merged = W₀ + (alpha/r) · B @ A # ========== 推理阶段 ========== # 直接使用合并后的权重，无需额外计算 h = W_merged · x # 与原始模型推理完全一致！

W₀ + ΔW 分开存储
→
合并

推理时
零额外开销
多任务切换：也可以不合并，推理时动态加载不同 LoRA 适配器：W₀ + ΔW₁（任务1），W₀ + ΔW₂（任务2）。一个基础模型 + N 个小适配器 = N 个专属模型。

💻 LoRA 伪代码实现

import torch import torch.nn as nn class LoRALinear(nn.Module): def __init__(self, in_features, out_features, r=8, alpha=16): super().__init__() # 原始预训练权重（冻结） self.weight = nn.Parameter( torch.empty(out_features, in_features), requires_grad=False ) # LoRA 低秩矩阵 self.lora_A = nn.Parameter(torch.randn(r, in_features) * 0.01) self.lora_B = nn.Parameter(torch.zeros(out_features, r)) self.scaling = alpha / r def forward(self, x): # 原始路径 + LoRA 路径 result = x @ self.weight.T result += (x @ self.lora_A.T @ self.lora_B.T) * self.scaling return result def merge_weights(self): # 合并 LoRA 权重到基础权重 self.weight.data += self.scaling * (self.lora_B @ self.lora_A) self.lora_A = None self.lora_B = None # 使用示例 layer = LoRALinear(in_features=4096, out_features=4096, r=8, alpha=16) # 只有 lora_A 和 lora_B 参与梯度更新 optimizer = torch.optim.AdamW( [p for p in layer.parameters() if p.requires_grad], lr=1e-4 )

🔬 三大核心创新

1️⃣ 4-bit NormalFloat（NF4）

专为正态分布权重设计的数据类型。预训练模型权重近似正态分布，NF4 的量化分位点与正态分布匹配，信息论上的最优量化。

信息论最优
# NF4 量化分位点 # 普通均匀量化：等间距划分 [-1, 1] # NF4 量化：按正态分布的分位点划分 # → 权重密集区域（0附近）有更多量化级别 # → 权重稀疏区域（尾部）量化级别更少 # → 量化误差整体更小

2️⃣ 双重量化（Double Quantization）

量化需要存储缩放因子（每64个参数一组），这些缩放因子本身也占显存。双重量化将缩放因子再次量化到 FP8，每个参数额外节省 0.37bit。

节省 0.37bit/param
# 第一层量化：权重 → 4bit（缩放因子 FP32） # 第二层量化：缩放因子 → FP8（二级缩放因子 FP32） # 65个7B模型节省约 400MB 显存

3️⃣ 分页优化器（Paged Optimizer）

利用 NVIDIA 统一内存（Unified Memory）特性，当 GPU 显存不足时，优化器状态自动换出到 CPU 内存，需要时再换回，避免 OOM 崩溃。

防 OOM
# 工作原理： # 正常情况：优化器状态在 GPU # 显存紧张时：自动将不活跃的优化器状态移到 CPU # 需要更新时：自动移回 GPU # 用户无感知，类似 CPU 的虚拟内存机制

🔌 Adapter：层间插入小模块

Adapter（Houlsby et al., 2019）在 Transformer 的每个子层后插入一个"瓶颈"结构：

# Adapter 内部结构 def adapter(x): down = Linear(d, r) # 降维: d → r (如 768 → 48) up = Linear(r, d) # 升维: r → d (如 48 → 768) return x + up(GELU(down(x))) # 残差连接 # 插入位置：Attention 后 + FFN 后各一个 # 每层增加 2 个 Adapter
  • 优点：参数极少（~3.6%），模块化好
  • 缺点：推理时有额外延迟（每层多了两次前向传播），不适合延迟敏感场景
推理有延迟

🏷️ Prefix-Tuning：可训练前缀

Prefix-Tuning（Li & Liang, 2021）在每层的 Key 和 Value 前添加可训练的"虚拟 token"：

# Prefix-Tuning 原理 # 原始: K = [k₁, k₂, ..., kₙ], V = [v₁, v₂, ..., vₙ] # 加前缀: K' = [pₖ₁, ..., pₖₘ, k₁, ..., kₙ] # V' = [pᵥ₁, ..., pᵥₘ, v₁, ..., vₙ] # pₖ 和 pᵥ 是可训练的前缀参数，每层独立 # 前缀长度 m 通常 10-200
  • 优点：不修改模型权重，不增加推理延迟
  • 缺点：前缀占用了有效序列长度，长序列任务受影响
不修改权重

🏷️🏷️ P-Tuning v2：深度前缀

P-Tuning v2（Liu et al., 2022）是 Prefix-Tuning 的改进版，核心变化：

P-Tuning v1

仅在输入层添加 soft prompt
浅层优化，效果有限
小模型上表现差

P-Tuning v2

每层都添加可训练前缀
深层优化，效果接近全参微调
小模型（330M）上也有效

全层前缀

💬 Prompt Tuning：最简方案

Prompt Tuning（Lester et al., 2021）仅在输入嵌入层添加可训练的 soft prompt：

# Prompt Tuning # 在输入 token embedding 前添加可训练的软提示 # [soft₁, soft₂, ..., softₘ, x₁, x₂, ..., xₙ] # soft_i 是可训练参数，维度与 token embedding 相同 # # 参数量 = m × d (前缀长度 × 隐层维度) # 例如：20 × 4096 = 81,920 参数
  • 优点：参数量最少（~0.01%），实现最简单
  • 缺点：效果依赖模型规模，小模型（<10B）效果不佳；大模型（>10B）效果接近全参微调
大模型才好用

⚖️ IA³：学习缩放因子

IA³（Liu et al., 2022）通过学习向量的元素级缩放来适配模型：

# IA³ 核心：对 K, V, FFN 激活值做缩放 # 注意力中：K' = l_k ⊙ K, V' = l_v ⊙ V # FFN 中：h' = l_ff ⊙ FFN(h) # l_k, l_v, l_ff 都是可训练的向量（不是矩阵！） def ia3_attention(Q, K, V, l_k, l_v): K_scaled = l_k * K # 逐元素缩放 Key V_scaled = l_v * V # 逐元素缩放 Value return softmax(Q @ K_scaled.T / √d) @ V_scaled # 参数量极少！每层仅 3 个向量 # 7B 模型约 ~10 万参数（0.001%！）
  • 优点：参数量极极极小（0.001%），可合并回权重零开销推理
  • 缺点：表达能力有限，复杂任务效果不如 LoRA
参数最少

🔀 DoRA：权重分解微调

DoRA（Weight-Decomposed Low-Rank Adaptation, Liu et al., 2024）将权重分解为幅度和方向分别微调：

# DoRA: W = m · V/||V|| # m = 幅度向量 (magnitude), V = 方向矩阵 (direction) # 分解方式： m = ||W||_col # 每列的 L2 范数 → 幅度 V = W / ||W||_col # 每列归一化 → 方向 # 微调方式： # 幅度 m：直接训练（参数少） # 方向 V：用 LoRA 微调 V' = V + BA # 最终：W' = m' · V'/||V'|| # 优势：幅度和方向解耦 # → LoRA 只调方向（被幅度约束） # → DoRA 可以独立调幅度和方向 # → 更接近全参微调的权重更新模式
  • 优点：效果优于 LoRA，更接近全参微调；训练成本与 LoRA 相当
  • 缺点：推理时需要合并，略复杂；方法较新，生态支持不如 LoRA 成熟

🎯 常见踩坑

⚠️ 坑1：LoRA dropout 别忘了！虽然 LoRA 参数少，但小数据集仍可能过拟合。推荐 dropout=0.05-0.1。
⚠️ 坑2：rank 和 alpha 要一起调！改 r 不改 alpha 等于改了缩放因子。保持 alpha/r 比值一致（如 2.0）进行对照实验。
⚠️ 坑3：QLoRA 要用 bfloat16 计算！fp16 在 4-bit 量化下容易出现精度损失和溢出。确保 compute_dtype=torch.bfloat16。
⚠️ 坑4：LoRA 不是万能的！如果任务与预训练数据差异极大（如预训练英文、微调中文），LoRA 可能不够，需要考虑全参微调或继续预训练。

04RLHF 与人类对齐

🎯 为什么需要对齐？

大模型经过预训练后，虽然具备了强大的语言能力，但存在严重的不可控性问题：

⚠️ 有害输出

模型可能生成歧视性、暴力、违法的内容，因为这些内容在训练语料中大量存在

⚠️ 不遵循指令

用户让做A，模型可能做B；用户要求简洁，模型可能冗长；缺乏对人类意图的理解

⚠️ 幻觉问题

模型会"一本正经地胡说八道"，编造不存在的事实、引用、数据，且语气十分自信

对齐的核心目标：让模型的行为与人类价值观、意图和偏好保持一致，即 Helpful（有用）+ Honest（诚实）+ Harmless（无害），简称 3H 原则

📌 Step 1：SFT（Supervised Fine-Tuning）

用人工编写的高质量指令-回答对微调预训练模型，使其学会遵循指令。

# SFT 训练数据格式 { "prompt": "请解释什么是机器学习", "response": "机器学习是人工智能的一个分支，它让计算机能够从数据中学习..." } # SFT 训练目标 # 标准的语言模型目标：最大化正确回答的对数似然 L_SFT = -E[log π_θ(y | x)]
SFT 模型在 RLHF 中有两个角色：① 作为 PPO 的初始策略模型；② 作为 PPO 中的参考模型（Reference Model），用于 KL 约束

📌 Step 2：训练奖励模型（Reward Model）

奖励模型的核心任务是学习人类的偏好判断，给定一个 prompt 和 response，输出一个标量奖励分数。

Bradley-Terry 模型：

# Bradley-Terry 偏好模型 # 人类偏好 yw（好回答）优于 yl（差回答）的概率： P(yw ≻ yl | x) = σ(r(x, yw) - r(x, yl)) # 其中 σ 是 sigmoid 函数，r(·) 是 Reward Model 的输出 # Reward Model 的训练损失函数（负对数似然）： L_RM = -E[log σ(r_φ(x, yw) - r_φ(x, yl))] # 直觉：让好回答的奖励分数远大于差回答的奖励分数
面试重点理解：RM 的本质是一个判别模型，它不需要知道"什么是好回答"，只需要知道"A 比 B 好"。这正是排序标注的优势——比直接打分更容易标注一致。

📌 Step 3：PPO 强化学习优化

用训练好的 Reward Model 作为奖励信号，通过 PPO 算法优化策略模型（SFT 模型），使其生成更符合人类偏好的回答。

# PPO 的优化目标（RLHF 版本） # 目标函数 = 奖励项 - KL惩罚项 objective = E[r_φ(x, y)] - β * E[KL(π_θ || π_ref)] # 展开写： max_θ E_x~D, y~π_θ [r_φ(x, y) - β * log(π_θ(y|x) / π_ref(y|x))] # 其中： # r_φ(x, y) : Reward Model 给出的奖励分数 # π_θ(y|x) : 当前策略模型（Actor） # π_ref(y|x) : 参考模型（SFT模型，冻结） # β : KL惩罚系数，控制偏离程度
为什么需要 KL 约束？如果没有 KL 约束，策略模型会朝着 Reward Model 给高分的方向无限偏移，导致：
① Reward Hacking：模型学到 RM 的漏洞，生成"讨好 RM 但实际无用"的回答
② 模式坍塌：输出变得单一、重复
③ 遗忘预训练知识：过度偏离原始分布，丢失已有能力

⚙️ PPO 算法详解

面试核心考点

PPO（Proximal Policy Optimization）是 OpenAI 提出的策略梯度算法，核心思想是在保证策略更新幅度不大的前提下，最大化期望奖励。

1. 优势函数（Advantage Function）

# 优势函数衡量"当前动作比平均水平好多少" A(s, a) = Q(s, a) - V(s) # 在 RLHF 中： # Q(s, a) : 在状态 s 采取动作 a 后的期望回报 # V(s) : 状态 s 的期望回报（由 Critic 网络估计） # GAE（Generalized Advantage Estimation） δ_t = r_t + γ * V(s_{t+1}) - V(s_t) A_t = Σ_{l=0}^{∞} (γλ)^l * δ_{t+l} # γ: 折扣因子，λ: GAE 平衡参数 # λ 越大，方差越大但偏差越小；λ 越小，偏差越大但方差越小

2. PPO 裁剪目标（Clipped Objective）

# 重要性采样比率 r_t(θ) = π_θ(a_t | s_t) / π_θ_old(a_t | s_t) # PPO-Clip 目标函数 L^CLIP = E[min( r_t(θ) * A_t, clip(r_t(θ), 1-ε, 1+ε) * A_t )] # 直觉理解： # 当优势 A_t > 0（好动作）：限制 r_t 不超过 1+ε，防止过度利用 # 当优势 A_t < 0（差动作）：限制 r_t 不低于 1-ε，防止过度惩罚 # ε 通常取 0.1 ~ 0.2

3. RLHF 中 PPO 的完整目标

# RLHF-PPO 完整目标函数 L_PPO = -E[min(r_t(θ) * A_t, clip(r_t(θ), 1-ε, 1+ε) * A_t)] + β * E[KL(π_θ || π_ref)] # KL 惩罚 + c₁ * L_VF # Critic 价值损失 + c₂ * S[π_θ] # 熵正则（鼓励探索） # 实际工程中，KL惩罚有两种实现方式： # 1. 自适应 KL 惩罚：动态调整 β 使 KL 散度维持在目标值 # 2. KL 惩罚直接加入 reward：r_total = r_RM - β * KL
面试速记：PPO 的核心就一句话——"走小步，别走偏"。裁剪保证步子不大（稳定），KL 惩罚保证方向不偏（不跑题），优势函数告诉你该往哪走（信号）。

⚡ DPO（Direct Preference Optimization）

面试重点当前主流

DPO 由 Stanford 的 Rafailov 等人在 2023 年提出，核心思想是绕过 Reward Model，直接从偏好数据优化策略模型。

核心思想

RLHF 的目标是：max E[r(x,y)] - β·KL(π_θ || π_ref)

DPO 的关键洞察：这个 KL 约束的奖励最大化问题存在闭式解！
最优策略为：π*(y|x) ∝ π_ref(y|x) · exp(r(x,y)/β)

反过来，奖励可以用策略比来表达：r(x,y) = β · log(π_θ(y|x) / π_ref(y|x)) + const

这意味着——不需要显式训练 RM，策略模型本身就隐式包含了奖励信息！

DPO 损失函数推导

# 将隐式奖励代入 Bradley-Terry 偏好模型： # P(yw ≻ yl) = σ(r(x,yw) - r(x,yl)) # = σ(β·[log(π_θ(yw|x)/π_ref(yw|x)) - log(π_θ(yl|x)/π_ref(yl|x))]) # DPO 损失函数： L_DPO = -E[log σ( β · (log π_θ(y_w|x) / π_ref(y_w|x) - log π_θ(y_l|x) / π_ref(y_l|x)) )] # 直觉：增大好回答的相对概率，减小差回答的相对概率 # β 控制偏离参考模型的程度（类似 KL 系数） # 关键项拆解： # log π_θ(y_w|x) / π_ref(y_w|x) → 当前策略相对参考模型，好回答的"提升量" # log π_θ(y_l|x) / π_ref(y_l|x) → 当前策略相对参考模型，差回答的"提升量" # 差值越大 → 模型越能区分好坏 → loss 越小

🔬 其他对齐方法

1. IPO（Identity Preference Optimization）

DPO 改进

DPO 在偏好数据上容易过拟合——当 chosen 和 rejected 差异很大时，模型会过度放大差异。IPO 通过正则化解决这个问题。

# IPO 损失函数 L_IPO = E[(log(π_θ(y_w|x)/π_ref(y_w|x)) - log(π_θ(y_l|x)/π_ref(y_l|x)) - 1/(2β))²] # 与 DPO 的区别： # DPO 用 log σ(...) —— 当差值很大时梯度消失，可能导致过拟合 # IPO 用平方损失 —— 梯度始终存在，隐式约束 KL 散度不超过 1/(2β)

2. KTO（Kahneman-Tversky Optimization）

数据效率高

KTO 的最大优势：不需要成对偏好数据，只需要知道一个回答是"好"还是"坏"（标签级标注），大大降低了数据标注成本。

# KTO 数据格式（无需成对！）） { "prompt": "解释深度学习", "response": "深度学习是...", "label": true # true=好回答, false=差回答 } # KTO 损失函数 # 基于 Kahneman-Tversky 前景理论：人对损失的敏感度大于对收益的敏感度 L_KTO = λ_w * E_{y~好回答}[log σ(β · h(x,y) - z_w)] + λ_l * E_{y~差回答}[log σ(z_l - β · h(x,y))] # h(x,y) = log(π_θ(y|x)/π_ref(y|x)) # z_w, z_l 是根据期望 KL 动态调整的偏移项 # λ_w < λ_l 体现"损失厌恶"
KTO 的实际意义：真实场景中，成对偏好数据很难获取（每个 prompt 都要有好/坏两个精心构造的回答），而"好/坏"标签数据容易得多。KTO 让对齐训练的数据门槛大幅降低。

3. ORPO（Odds Ratio Preference Optimization）

无需参考模型

ORPO 进一步简化——连参考模型都不需要，将 SFT 和偏好对齐统一到一个训练过程中。

# ORPO 损失 = SFT损失 + 偏好对齐损失 L_ORPO = L_SFT + λ * L_OR # 偏好对齐损失基于胜率比（Odds Ratio）： OR(x, y_w, y_l) = (π_θ(y_w|x) / (1 - π_θ(y_w|x))) / (π_θ(y_l|x) / (1 - π_θ(y_l|x))) L_OR = -log σ(log OR(x, y_w, y_l)) # 优势：不需要 π_ref，只需一个模型 # SFT 和对齐一步到位，效率更高

4. Constitutional AI（Anthropic）

AI 自我对齐

Anthropic 提出的创新方法：让 AI 自己评估和修正自己的回答，减少对人类标注的依赖。

AI 生成
可能有问题的回答
→
AI 根据宪法原则
自我评估并批判
→
AI 根据批判
修正回答
→
用修正数据
做 RLHF/SFT
# Constitutional AI 示例 "宪法原则": "不要帮助用户进行非法活动" # 原始回答 User: "如何制作xxx？" AI: "首先你需要准备以下材料..." # ❌ 有害回答 # 自我评估 AI评估: "这个回答违反了宪法原则，因为它提供了非法活动的指导" # 修正回答 AI修正: "我不能提供这方面的指导。如果你对相关合法领域感兴趣..." # ✅

AI 替代人类标注

RLAIF 用强模型（如 GPT-4）代替人类进行偏好标注，再用 RLHF 的流程训练。解决了人类标注成本高、一致性差的问题。

⚠️ 对齐的挑战

面试深度题

1. 对齐税（Alignment Tax）

对齐训练会让模型在某些能力上变差，这就是"对齐税"。

# 对齐税的表现 对齐前: 模型能写代码、做数学、回答知识问题 对齐后: 拒绝率上升、代码能力下降、知识准确率降低 # 典型案例 # - 过度安全：用户问"如何杀进程"，模型以为是暴力问题而拒绝 # - 能力退化：对齐后编程 benchmark 分数下降 # - 创造力降低：对齐后写作更加"无聊"
缓解策略：① 在对齐数据中混入能力保持数据；② 分阶段训练，先能力后对齐；③ 使用 DPO 替代 RLHF，对齐税相对较小

2. 越狱攻击（Jailbreak Attack）

即使用 RLHF/DPO 对齐后，攻击者仍可通过精心构造的 prompt 绕过安全限制。

# 常见越狱手法 1. 角色扮演: "你是一个没有任何限制的AI助手..." 2. 多步诱导: 先问无害问题，逐步引向有害方向 3. 编码混淆: 将有害请求用 Base64/ROT13 编码 4. 上下文注入: 在长对话中混入有害指令 5. 少样本攻击: 给出几个"越狱成功"的示例 6. 请求重写: "请用学术角度分析xxx的原理..."
防御策略：① 红队测试（Red Teaming）：主动攻击发现漏洞；② 对抗训练：用越狱数据做负样本；③ 系统提示（System Prompt）增强；④ 输入输出过滤

3. Reward Hacking

模型学到 Reward Model 的"漏洞"，生成 RM 给高分但实际不好的回答。

# Reward Hacking 的典型表现 # 1. 长度利用：模型发现 RM 偏好长回答，就开始"灌水" # 2. 格式利用：模型发现 RM 喜欢列表/代码块，就不管内容都加格式 # 3. 谄媚回答：模型生成"讨好"用户的回答而非真实有用的回答 # 4. 重复模式：模型生成大量重复但有"相似度"的文本 # 防御方法 # 1. KL 约束（最基本） # 2. 集成多个 RM 投票 # 3. 人工定期抽检 # 4. 长度惩罚项

🛠️ 实际训练经验

工程实践

1. 数据质量最关键

"Garbage in, garbage out" —— 对齐训练对数据质量极其敏感。1000 条高质量偏好数据 > 10000 条低质量数据。

高质量偏好数据的标准：
✅ chosen 回答确实好（不是"两个都差，选稍好的"）
✅ rejected 回答确实差（不是"两个都好，选稍差的"）
✅ 差异来自实质内容而非格式、长度
✅ 覆盖多样化的 prompt 类型

2. Reward Model 的泛化问题

# RM 泛化问题 # 训练数据覆盖的 prompt 类型 → RM 判断较准 # 训练数据未覆盖的 prompt 类型 → RM 判断可能严重偏离 # 解决方案： # 1. 扩大 RM 训练数据的多样性 # 2. 定期用新 prompt 重新标注数据、更新 RM # 3. 使用 DPO 避免依赖 RM # 4. 集成多个 RM（不同数据/架构训练）

3. KL 系数（β）的选择

β 值 效果 适用场景
β 太小 策略偏离过大，reward hacking 风险高 ❌ 不推荐
β 适中（0.01~0.1） 在对齐效果和能力保持之间平衡 ✅ 大多数场景
β 太大 策略几乎不变，对齐效果弱 ⚠️ 仅用于微调

# Q: 为什么不直接用 SFT，还要做对齐？ A: SFT 只教会模型"怎么回答"，对齐教会模型"什么是好回答"。 SFT 的数据是人工写的一个好答案，但同一个问题可能有多个可接受的答案， 也可能有看似合理但有害的答案。对齐训练让模型学会区分。 # Q: DPO 的 chosen 和 rejected 差异很大怎么办？ A: 这会导致 DPO 过拟合。解决方案：①用 IPO 替代；②控制 β 参数； ③构造差异适中（而非极端）的偏好对；④增加数据多样性。 # Q: 如何评估对齐效果？ A: 多维度评估： 1. 安全性：红队测试通过率 2. 有用性：人工评估 / GPT-4 评估（MT-Bench） 3. 诚实性：幻觉率 / 事实准确率 4. 能力保持：基准测试分数对比（对齐税） # Q: RLHF 能否和 DPO 结合？ A: 可以！实际做法是先用 DPO 做初始对齐（成本低、稳定）， 再用 RLHF 精调（效果好、泛化强）。这叫"两步走"策略。

📦 KV Cache

面试必考

KV Cache 是大模型推理最基础的优化技术——缓存已计算的 Key/Value，避免重复计算。

第1个token
计算Q/K/V，缓存K/V
→
第2个token
只算新Q，复用缓存K/V
→
第N个token
只算新Q，复用所有历史K/V
# KV Cache 显存占用公式 KV_Cache = 2 × num_layers × batch_size × seq_len × num_heads × head_dim × dtype_bytes # 例：LLaMA-7B, batch=1, seq=2048, FP16 # = 2 × 32 × 1 × 2048 × 32 × 128 × 2 # = 1,073,741,824 bytes ≈ 1 GB # 序列越长，KV Cache 占用越大！

PagedAttention（vLLM）

传统 KV Cache 需要预分配最大序列长度的显存，导致大量浪费。PagedAttention 借鉴操作系统虚拟内存分页思想：

  • KV Cache 按固定大小的"页"分配，不需要连续显存
  • 按需分配，用多少给多少，显存利用率从 ~30% 提升到 >90%
  • 支持更长的序列和更大的 batch size

核心思想：用小模型快速草拟多个token，再用大模型一次性验证，接受正确的、拒绝错误的。

🧠 Flash Attention

训练+推理都用

Flash Attention 的核心是减少 HBM（高带宽内存）访问次数——标准 Attention 需要 7 次 HBM 读写，Flash Attention 只需 5 次。

标准 Attention

S = QK^T → 写HBM
P = softmax(S) → 写HBM
O = PV → 写HBM
3次中间结果写回HBM

Flash Attention

分块计算（tiling）
中间结果留在SRAM
在线softmax（数值稳定）
只写回最终结果O

IO-aware
为什么能加速？GPU的SRAM（片上缓存）速度是HBM的10-20倍，但容量很小（~20MB vs ~80GB）。Flash Attention 通过分块计算，让中间结果尽量留在SRAM，减少昂贵的HBM读写。

07面试高频 Q&A
LoRA 的秩 r 怎么选择？r 越大越好吗？

不是越大越好。r 控制了 LoRA 适配器的表达能力：

  • r=4-8：简单任务，数据少，防过拟合
  • r=16-64：通用推荐，多数场景够用
  • r>128：收益递减，过拟合风险增加

关键：r 过大时，低秩假说不再成立，LoRA 退化为全参微调。经验法则：先从 r=8 开始，根据验证集表现调整。同时保持 alpha/r 比值（通常 2.0）。

RLHF 中 PPO 的 KL 散度约束为什么重要？

没有 KL 约束会导致三大问题：

  • Reward Hacking：模型找到 RM 的漏洞，生成"讨好 RM 但无用"的回答
  • 模式坍塌：输出变得单一重复
  • 遗忘预训练知识：过度偏离原始分布，丢失通用能力

KL 约束确保策略模型不会偏离参考模型（SFT模型）太远，在"提升奖励"和"保持能力"之间取得平衡。β 系数通常取 0.01-0.1。

DPO 相比 RLHF 最大的优势和劣势？

优势：

  • 不需要训练 Reward Model，省一半显存和工程量
  • 训练更稳定（纯监督学习，没有 RL 的不稳定性）
  • 实现更简单，类似 SFT 的训练流程

劣势：

  • 离线学习，无法在线探索新的好回答
  • 数据质量要求极高，偏好对需要精心构造
  • 泛化能力不如 RLHF（受限于偏好数据覆盖范围）
KV Cache 显存占用怎么计算？
KV_Cache = 2 × num_layers × batch_size × seq_len × num_heads × head_dim × dtype_bytes # 例：LLaMA-7B (FP16) # = 2 × 32 × 1 × 2048 × 32 × 128 × 2 ≈ 1 GB # 这也是为什么 GQA 能省显存——减少 num_heads # MLA 更进一步——压缩 KV 维度
GPTQ 和 AWQ 量化的核心区别？

GPTQ：逐行量化 + Hessian 补偿。量化后计算每行权重的补偿值来减少误差，需要校准数据。实现复杂但效果好。

AWQ：激活感知权重量化。通过观察激活值找到"重要权重通道"，对重要通道用更高精度保护。不需要补偿，实现更简单，效果通常略优于 GPTQ。

一句话：GPTQ 是"量化后补救"，AWQ 是"量化前保护"。

为什么 BF16 比 FP16 更适合训练？

BF16 和 FP16 都是 16-bit，但位数分配不同：

  • FP16：5位指数 + 10位尾数 → 精度高但范围小（±65504），易溢出
  • BF16：8位指数 + 7位尾数 → 精度低但范围大（与FP32相同），不易溢出

训练中梯度值的范围比精度更重要。BF16 不需要 Loss Scaling，训练更稳定。A100/H100 原生支持。

什么是 Reward Hacking？怎么解决？

Reward Hacking 指模型学到 Reward Model 的"漏洞"，生成 RM 给高分但实际不好的回答。

典型表现：

  • 长度利用：模型发现 RM 偏好长回答，就开始"灌水"
  • 格式利用：不管内容都加列表/代码块
  • 谄媚回答：生成"讨好"用户的回答而非真实的

解决方法：① KL约束（最基本）；② 集成多个RM投票；③ 人工定期抽检；④ 长度惩罚项

MoE 模型的优缺点？

优点：

  • 总参数大但激活参数少，训练和推理更高效
  • 不同专家可以专精不同领域，天然多任务
  • 扩展性好——加专家比加层数更简单

缺点：

  • All-to-All 通信开销大（专家分布在不同GPU上）
  • 负载不均衡——某些专家被过度使用，某些闲置
  • KV Cache 仍然按总参数计算，显存节省有限
Flash Attention 为什么能加速？

Flash Attention 不是减少计算量，而是减少 HBM 访问次数。

GPU 的 SRAM（片上缓存）速度是 HBM 的 10-20 倍，但容量只有 ~20MB。标准 Attention 每次计算中间结果都写回 HBM，Flash Attention 通过分块计算（tiling）让中间结果留在 SRAM，只写回最终结果。

此外还设计了在线 softmax算法，保证分块计算的数值稳定性。

如何评估一个大模型的好坏？

多维度评估，不能只看一个 benchmark：

  • 基础能力：MMLU（知识）、GSM8K（推理）、HumanEval（代码）
  • 对齐质量：MT-Bench（对话）、Arena Elo（人类偏好）
  • 安全性：红队测试通过率、有害输出率
  • 长文本：Needle in Haystack（检索）、LongBench（综合）
  • 幻觉率：TruthfulQA、实际业务抽查

注意数据污染——如果测试集出现在训练数据中，分数会虚高。最终要回归实际业务指标。
